ซอฟต์แวร์คืออะไร ? (What is Software?)

นิยามแบบเข้าใจง่าย

  • ชุดของ คำสั่ง (programs) ที่สั่งให้คอมพิวเตอร์ทำงานเฉพาะอย่าง
  • รวมถึง ข้อมูล (data), เอกสาร (documentation), และ ขั้นตอนการปฏิบัติงาน (operating procedures) ที่เกี่ยวข้อง
┌────────────── Software ──────────────┐
│  Programs (โค้ด, executable)        │
│+ Data (ไฟล์ข้อมูล, config)          │
│+ Documentation (คู่มือ, SRS, ฯลฯ)   │
│+ Operating Procedures (วิธีใช้งาน)  │
└──────────────────────────────────────┘

ใช้ภาพกล่องนี้อธิบายว่า “ซอฟต์แวร์” ไม่ใช่แค่ไฟล์ .exe หรือโค้ด แต่มีส่วนอื่น ๆ ที่ต้องดูแลด้วย

ตัวอย่างรอบตัวเรา

System Software
  • Windows, Linux, Android, iOS
  • Compiler, Runtime, Drivers
Application Software
  • LINE, Facebook, Shopee, Grab
  • VS Code, Figma, Chrome

มุมมองสำคัญสำหรับวิศวกรซอฟต์แวร์

เราไม่ได้แค่ “เขียนโค้ด” แต่ต้องคิดถึงข้อมูล, เอกสาร และวิธีการใช้งาน/ดูแลระบบ เพื่อให้ระบบอยู่ได้ในโลกจริงนาน ๆ

องค์ประกอบของซอฟต์แวร์

Software Components (Program • Data • Docs • Procedures)

องค์ประกอบหลัก

  • Programs – โค้ดที่รันบนเครื่อง / server / cloud
  • Data – ฐานข้อมูล, ไฟล์, configuration, logs
  • Documentation – SRS, Design docs, คู่มือใช้งาน, API docs
  • Operating Procedures – วิธี deploy, วิธี backup, วิธี handle incident, SOP ต่าง ๆ
Infographic: 4 วงกลมซ้อนกัน – Program, Data, Docs, Procedures = Software System

ใช้ภาพ 4 วงกลม / 4 blocks แสดงว่า ถ้าขาดอย่างใดอย่างหนึ่ง คุณภาพของระบบจะลดลงทันที

แยกความหมายให้ชัด

Program vs Software vs Software System

Program
  = โค้ดชิ้นหนึ่งที่ทำงานเฉพาะงาน
    (เช่น โปรแกรมคำนวณเกรดเล็ก ๆ)

Software
  = Program + Data + Docs + Procedures
    (พร้อมใช้งานจริงในองค์กร/หน่วยงาน)

Software System
  = Software หลายส่วนเชื่อมกัน
    + Database หลายตัว
    + External Services / API
    + Infra / Network / Cloud

จากโค้ด → วิศวกรรม

What is Software Engineering?

“การใช้หลักการทางวิศวกรรม กระบวนการที่มีวินัย และเครื่องมือที่เหมาะสม ในการพัฒนา ดูแล และบริหารซอฟต์แวร์ ให้ตอบโจทย์ผู้ใช้และคุณภาพที่ต้องการ”

ไม่ใช่แค่ “เขียนโปรแกรม”

  • สนใจทั้ง วงจรชีวิตซอฟต์แวร์ ไม่ใช่แค่ตอนเขียนโค้ด
  • ต้องคิดเรื่อง คุณภาพ (Quality Attributes) เช่น Performance, Security, Maintainability
  • คำนึงถึงทีมหลายบทบาท: Dev, QA, Ops, Business, Users
Code-centric mindset
  → "เขียนให้รันได้"

SE mindset
  → "ออกแบบให้โตได้, แก้ได้, อยู่ได้ยาว"

คำถามชวนคิด

  • ระบบที่ดี “เฉพาะวันนี้” กับ “อยู่ได้ 5–10 ปี” ต่างกันอย่างไร?
  • ถ้าทีมใหม่เข้ามารับช่วงต่อ จะอ่านโค้ด/เอกสารแล้วเข้าใจไหม?
  • ถ้าผู้ใช้โตจาก 100 → 100,000 คน ระบบยังรอดไหม?

ตารางเปรียบเทียบ (จาก Lecture)

Programming vs Software Engineering

Programming
  • โฟกัสที่ “เขียนโค้ดให้รันได้” ตามโจทย์เล็ก ๆ
  • มักคิดทีละไฟล์ / ทีละฟังก์ชัน
  • ไม่ค่อยสนใจเรื่อง deploy, scaling, maintenance
  • เอกสาร/โครงกระบวนการมีน้อยหรือไม่มีเลย
Software Engineering
  • โฟกัสทั้ง “ทำงานได้” + “ทำงานได้ดีระยะยาว”
  • คิดเป็นระบบ: Architecture, Modules, Interfaces
  • สนใจ deployment, monitoring, backup, incident handling
  • มี process, docs, standard, review, testing ครบวงจร

ใน ENGCE301 เราจะเริ่มขยับจากฝั่ง Programming → Software Engineering ผ่านทั้งทฤษฎี (SDLC, Process Models) และการลงมือทำใน Term Project

Software Crisis & ทำไมต้องมี Software Engineering

Software Crisis คืออะไร?

  • โครงการซอฟต์แวร์จำนวนมาก ส่งไม่ทัน / เกินงบ / ใช้งานจริงไม่ได้
  • ระบบในโดเมนสำคัญ (การแพทย์ การบิน การเงิน) เกิดบั๊กจนเกิดความเสียหายใหญ่
  • ขนาดระบบใหญ่เกินกว่าที่ทีมเล็ก ๆ จะ “จำทุกอย่างในหัว” ได้

สถิติจาก Lecture

68% ของโครงการล้มเหลวเพราะ requirement ไม่ชัดเจน
การเปลี่ยน requirement ช่วงท้าย → ต้นทุนสูงขึ้น > 100 เท่า
โครงการที่จัดการ requirement ดี มีอัตราสำเร็จสูงอย่างมีนัยสำคัญ

Software Engineering เข้ามาช่วยอย่างไร?

ก่อนหน้า:
  "เก่งเขียนโค้ด = ทำโปรเจกต์ใหญ่ได้"

ผลลัพธ์:
  • ระบบซับซ้อนเกินควบคุม
  • แก้บั๊กที = พังอีกหลายส่วน
  • ทีมใหม่เข้ามา → อ่านโค้ดไม่รู้เรื่อง

คำตอบ:
  นำ "ความเป็นวิศวกรรม" เข้ามาช่วย
  = หลักการ + กระบวนการ + มาตรฐาน

SWEBOK, ISO/IEC/IEEE 12207 & ACM/IEEE

SWEBOK (Software Engineering Body of Knowledge)

  • สรุป “องค์ความรู้หลัก” ที่วิศวกรซอฟต์แวร์ควรรู้
  • มีหลาย knowledge areas เช่น Requirements, Design, Construction, Testing, Maintenance ฯลฯ
  • ใช้เป็น reference สำหรับหลักสูตรมหาวิทยาลัย และ industry

สำหรับนักศึกษา

วิชา ENGCE301 ครอบคลุมหลายส่วนของ SWEBOK โดยเฉพาะด้าน Requirements, Design, Construction, Process, Tools & Methods

ISO/IEC/IEEE 12207 & ACM/IEEE Curriculum

  • ISO/IEC/IEEE 12207 – มาตรฐานกระบวนการวงจรชีวิตซอฟต์แวร์ (Software Life Cycle Processes)
  • ACM/IEEE Curriculum – แนวทางออกแบบหลักสูตรวิศวกรรมซอฟต์แวร์ในระดับมหาวิทยาลัย

  • การเปลี่ยน requirement กลายเป็นเรื่องปกติ → ต้องใช้ process ที่รองรับการเปลี่ยน (เช่น Agile)

SDLC – ภาพรวมวงจรชีวิตซอฟต์แวร์

SDLC: Software Development Life Cycle

คำนิยามในวิชา:

SDLC คือกรอบขั้นตอน/ช่วงเวลาที่ใช้จัดระเบียบการพัฒนาซอฟต์แวร์ ตั้งแต่คิดไอเดีย → เก็บ requirement → ออกแบบ → พัฒนา → ทดสอบ → นำไปใช้ → บำรุงรักษา 

SDLC (6 ขั้นตอนหลัก – เวอร์ชันวิชานี้)

 IDEA → REQUIREMENTS → DESIGN → IMPLEMENTATION
        ↓            ↓
      TESTING    DEPLOYMENT
        ↓
   MAINTENANCE / EVOLUTION

ทุกกิจกรรมในวิชา ENGCE301 จะผูกกับอย่างน้อย 1 ขั้นของ SDLC เสมอ

ทำไมต้องมี SDLC?

  • ระบุขั้นตอนให้ชัด → ทุกคนในทีมเห็นภาพรวมเดียวกัน
  • ช่วยวางแผนและติดตามความคืบหน้าโครงการ
  • ลดโอกาสลืมกิจกรรมสำคัญ เช่น การทดสอบ, เอกสาร, การส่งมอบ

ข้อควรจำ

SDLC เป็น “กรอบคิด” ที่สามารถใช้กับ process model หลายแบบ เช่น Waterfall, Iterative, Incremental, RAD, Spiral, Agile ฯลฯ

ขั้นตอน SDLC (เวอร์ชันเข้าใจง่าย)

SDLC 6 ขั้นตอน – Step-by-Step

  • 1. Requirements – เข้าใจปัญหา, ผู้ใช้, และขอบเขต
  • 2. Design – ออกแบบโครงสร้างระบบ, UI คร่าว ๆ, Database, API
  • 3. Implementation – เขียนโค้ด, เขียน test ขั้นต้น
  • 4. Testing – ตรวจสอบว่าระบบทำงานตรงตาม requirement
  • 5. Deployment – นำระบบขึ้นใช้งานจริง (หรือ demo environment)
  • 6. Maintenance – แก้บั๊ก, ปรับปรุง, เพิ่มฟีเจอร์ในอนาคต
Infographic: วงกลม 6 ชิ้น หรือขั้นบันได 6 ขั้น พร้อมไอคอนแต่ละขั้น (Req, Design, Code, Test, Deploy, Maintain)

ใน Term Project – จะวนผ่าน 6 ขั้นนี้แบบสเกลเล็ก ๆ หลายรอบ เพื่อฝึกใช้ SDLC ในสถานการณ์จริง

แนวเปรียบเทียบ (แบบย่อ)

  • Waterfall / V-Model – เน้นเอกสาร, ขั้นตอนชัด, เหมาะกับงานข้อกำหนดนิ่ง
  • Incremental / Iterative – ทำทีละส่วน, เพิ่มฟีเจอร์เป็นรอบ ๆ
  • RAD / Prototyping – เน้นทำ Prototype คุยกับผู้ใช้เร็ว ๆ ก่อนลงรายละเอียด
  • Spiral – เน้นวิเคราะห์ความเสี่ยง (risk) เป็นรอบ ๆ
  • Agile, Scrum, Kanban – เน้นปรับตัว, feedback บ่อย, team communication

Waterfall Model

  • ทำงานทีละเฟส: Requirements → Design → Implementation → Testing → Deployment → Maintenance
  • ข้อดี: ขั้นตอนชัด, เอกสารครบ, เหมาะกับข้อกำหนดค่อนข้างนิ่ง
  • ข้อจำกัด: Feedback จากผู้ใช้มักมาช่วงท้าย, เปลี่ยน requirement ยาก
Infographic: Staircase Waterfall (ตาม lecture)

V-Model

  • ขยายจาก Waterfall โดยจับคู่แต่ละเฟสกับระดับของการทดสอบ
  • เช่น System Requirements ↔ System Testing, Design ↔ Integration Testing ฯลฯ
  • เน้นความสัมพันธ์ระหว่าง “สิ่งที่ออกแบบ” กับ “สิ่งที่ทดสอบ”
Infographic: ตัว V – ด้านซ้าย = Development, ด้านขวา = Testing Levels

Incremental, Iterative, RAD & Prototyping

Incremental & Iterative

  • Incremental – แบ่งระบบเป็น “ส่วนย่อย (increments)” แล้วส่งมอบทีละส่วน
  • Iterative – ทำเวอร์ชันแรกเร็ว ๆ แล้วปรับปรุงจาก feedback หลาย ๆ รอบ
  • มักใช้ร่วมกัน: วนรอบ (iterate) บน increments ต่าง ๆ
Infographic: หลายวงกลม/กล่องต่อกัน – Version 1 → 2 → 3 พร้อมฟีเจอร์เพิ่ม

RAD & Prototyping

  • RAD (Rapid Application Development) – เน้นสร้างระบบเร็ว ๆ โดยใช้ tools/component reuse
  • Prototyping Model – ทำ Prototype เพื่อคุยกับผู้ใช้ก่อนลงรายละเอียดจริง
  • เหมาะกับ requirement ยังไม่ชัด แต่ต้องการเห็นภาพระบบเร็ว ๆ
Infographic: Loop “Quick Plan → Prototype → Feedback → Refine”

Spiral Model (Risk-driven)

แนวคิดหลักของ Spiral

  • แบ่งโครงการเป็น “รอบ (loops)” ซ้อนเป็นรูปเกลียว
  • แต่ละรอบมี 4 ส่วนหลัก: Objective setting, Risk assessment, Development & validation, Planning
  • เหมาะกับโครงการใหญ่/ซับซ้อน/ความเสี่ยงสูง
Infographic: วงเกลียว 4 ส่วน (จาก lecture – Spiral Model)

ในวิชานี้จะพูดถึง Spiral เพื่อให้เห็นภาพ “risk-driven model” แม้ Term Project จะไม่ได้ใช้เต็มรูปแบบ

Agile Mindset & วิธีคิด

Agile Model – ค่านิยมสำคัญ

Agile Values (จาก Agile Manifesto)

  • Individuals & Interactions > Processes & Tools
  • Working Software > Comprehensive Documentation
  • Customer Collaboration > Contract Negotiation
  • Responding to Change > Following a Plan
ตีความสำหรับนักศึกษา:
  • พูดคุยกันบ่อย ๆ ในทีมสำคัญกว่ากระดาษเยอะ ๆ
  • ส่งมอบงานทีละชิ้นเล็ก ๆ → ดู feedback ทันที
  • เอกสารยังจำเป็น แต่ให้ "พอเหมาะ"

ใน ENGCE301

  • ระดับ “เทอม” ใช้โครงแบบ Stage (คล้าย Waterfall)
  • ระดับ “สัปดาห์/งานย่อย” ใช้แนวคิด Agile (Kanban, feedback สั้น ๆ)
  • เป้าหมายคือให้นักศึกษาสัมผัส Agile ในสเกลที่ทำได้จริงในรายวิชา

Agile Frameworks

Scrum & Kanban – ภาพเดียวเข้าใจง่าย

Scrum – วงรอบการทำงานแบบ Sprint

  • แบ่งงานเป็นรอบสั้น ๆ (Sprint) 1–4 สัปดาห์
  • มี ceremony หลัก: Sprint Planning, Daily Scrum, Sprint Review, Retrospective
  • บทบาทหลัก: Product Owner, Scrum Master, Development Team

บทบาทในทีม & Value Chain

บทบาทหลักในทีมซอฟต์แวร์

  • Software Developer / Engineer – เขียนโค้ด, ทดสอบเบื้องต้น, แก้บั๊ก
  • Requirements Engineer / BA – คุยกับ “ลูกค้า/ผู้ใช้ (จำลอง)” แล้วแปลงเป็น use case / user story / SRS ย่อ
  • Software Architect (เบื้องต้น) – คิดโครงสร้างระบบ (frontend–backend–DB), API คร่าว ๆ
  • QA Engineer / Tester – ช่วยออกแบบ test case และทดสอบ feature ที่ทีมพัฒนา
  • DevOps (เบื้องต้น) – ดูเรื่องการรันระบบในเครื่อง dev / demo, script, npm command ต่าง ๆ
